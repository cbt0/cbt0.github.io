# CBT 문제풀이 사이트 제작 계획서 (Plan)

이 계획서는 **최강 CBT** 스타일의 자격증 기출문제 학습 사이트인 `cbt0.github.io`를 구축하기 위한 상세 계획입니다. `gas_CBT_2017_2025_final.docx` 파일에서 가스기능사 2017년~2025년(총 18회차, 1080문제) 데이터를 완벽하게 추출 완료했으며, 이를 바탕으로 미려하고 모던한 웹사이트를 제작합니다.

---

## 1. 프로젝트 개요
- **목표**: 에너지/가스/공조 분야 기출문제를 풀 수 있는 최강 CBT 스타일의 반응형 웹 어플리케이션 구축
- **핵심 기술**: HTML5, Vanilla CSS (Premium Dark Mode & Glassmorphism), JavaScript (ES6+)
- **배포 환경**: GitHub Pages (`https://cbt0.github.io`)
- **데이터소스**: `gas_CBT_2017_2025_final.docx` (파싱 완료 -> `gas_questions.json`)

---

## 2. 웹사이트 메뉴 구성
웹사이트는 상단/측면 네비게이션을 통해 아래 메뉴를 제공합니다:

1. **소개말 (Home)**
   - 사이트 소개, 학습 가이드, 전체 학습 통계 (총 푼 문제 수, 정답률 등)
2. **가스기능사 (실제 데이터 제공)**
   - 2017년 1회 ~ 2025년 3회 (총 18회차, 회차별 60문제)
3. **에너지기능사 (준비중/샘플 기출)**
   - 학습 인프라 테스트용 샘플 문제 제공 및 준비중 안내
4. **에너지산업관리기사 (준비중/샘플 기출)**
   - 샘플 문제 제공 및 준비중 안내
5. **에너지기능장 (준비중/샘플 기출)**
   - 샘플 문제 제공 및 준비중 안내
6. **공조기능사 (준비중/샘플 기출)**
   - 샘플 문제 제공 및 준비중 안내

## 3. 핵심 기능 요구사항
- **회차 선택 화면**: 각 과목 클릭 시 연도/회차별 리스트가 카드 형태로 표시되며, 기존 풀이 기록(학습 진도 및 최고 점수)을 로컬 스토리지(`localStorage`)에서 불러와 시각적으로 표시.
- **CBT 풀이 화면**:
  - **기본 레이아웃**: 좌측 문제 네비게이터(1~60번 마킹판), 우측 현재 문제 및 보기 화면.
  - **힌트 버튼**: 문제별 해설/힌트를 볼 수 있는 버튼 제공.
  - **실시간 채점 및 자동 힌트**: 번호(보기 ①~④)를 클릭하는 즉시 실시간으로 채점을 진행하여 정답/오답 표시를 하고, 해설(힌트)을 자동으로 아래에 펼쳐줌.
  - **전체 채점**: 60문제를 모두 풀거나 중간에 제출 버튼을 누르면 시험을 종료하고 최종 점수(예: 45/60, 75점)와 **합격/불합격 여부**(60점/36문제 이상 합격)를 미려한 모달창으로 출력.

## 4. UI/UX 디자인 시스템 (Premium UI)
사용자를 압도할 수 있는 프리미엄 디자인을 적용합니다.
- **색상 팔레트 (Sleek Dark Mode & Violet Gold Hues)**:
  - 배경: `deep dark gray/slate` (#0f172a, #1e293b)
  - 액센트: `indigo violet` (#6366f1) & `emerald green` (#10b981, 정답 표시) & `rose red` (#f43f5e, 오답 표시)
  - 텍스트: `pure white` (#f8fafc) & `gray text` (#94a3b8)
- **타이포그래피**: Google Fonts의 `Inter` 및 `Noto Sans KR` 조합으로 가독성 최적화.
- **컴포넌트 스타일**:
  - 반투명 유리 효과(Glassmorphism)와 입체적인 그림자 적용.
  - 보기 클릭, 버튼 호버 시 부드러운 트랜지션 애니메이션 (`transition: all 0.3s ease`).
  - 모바일에서도 한눈에 보이는 반응형 레이아웃 (Media Queries 적극 활용).

- **파싱 프로그램**: `parse_docx_refined.py`를 활용하여 XML 구조 완벽 해독.

- **산출물**: `gas_questions.json`

## 6. 개발 단계별 구현 계획
1. **1단계**: 데이터 파싱 및 검증 (완료 -> `gas_questions.json` 생성됨)
2. **2단계**: 기본 폴더 구조 및 파일 생성 (`index.html`, `style.css`, `app.js`)
3. **3단계**: 디자인 시스템 정의 및 CSS 작업 (Sleek Dark Mode, 반응형 그리드)
4. **4단계**: SPA 라우팅 및 과목/회차 선택 화면 연동
5. **5단계**: 실시간 채점 및 힌트 자동 활성화 기능 구현
6. **6단계**: 다 푼 후 전체 채점 결과 모달창 및 로컬 스토리지 진도 저장 기능 탑재

## 7. 데이터 파일 구조 변경 검토 (단일 JSON vs 회차별 분리)

사용자의 질의("기출문제를 하나의 파일 `gas_questions.json`으로 묶어둔 것에 대해, 회차별로 쪼개어 보관하는 것이 더 편리하지 않을까?")에 대한 아키텍처 관점의 검토 의견입니다.

### 1) 단일 파일 구조 (Current: `gas_questions.json`)
* **장점**:
  - **네트워크 오버헤드 감소**: 첫 화면 진입 시 또는 학습을 시작할 때 약 1.1MB의 JSON 데이터를 단 1회의 HTTP Request로 로드하므로, 이후 화면 전환 및 회차 로딩 시 지연 시간(Latency)이 0ms에 가깝습니다.
  - **통합 검색 및 데이터 가공 용이성**: 추후 "전체 기출문제 대상 랜덤 60문제 모의고사", "키워드 통합 검색", "과목별 취약 유형 분석" 등의 분석 및 추출 기능을 자바스크립트로 구현하기에 매우 직관적이고 용이합니다.
  - **클라이언트 소스 코드의 단순화**: 퀴즈 엔진이 여러 번의 비동기 통신을 핸들링할 필요가 없으므로 상태 관리가 쉽고 버그 발생율이 줄어듭니다.
* **단점**:
  - **초기 로딩 시간**: 네트워크 속도가 3G 수준으로 매우 느린 특수 환경에서는 첫 로딩 시 수 초간 딜레이가 발생할 수 있습니다.
  - **유지보수 작업의 위험성**: 특정 회차의 오타를 수정하거나 포맷을 고칠 때 대형 JSON(1만여 라인)을 건드려야 하므로 실수로 인해 전체 JSON이 깨질 수 있습니다.

### 2) 회차별 분리 구조 (Alternative: `gas_2017_1.json`, `gas_2017_3.json`, ...)
* **장점**:
  - **초기 로딩 최소화**: 사용자가 선택한 특정 회차의 60문제(약 60KB 내외)만 즉각 `fetch()`해 오므로 초기 구동 성능이 비약적으로 향상됩니다.
  - **데이터 관리의 안전성**: 문제 수정이나 회차 추가가 극도로 간편합니다. 기존 전체 파일의 문법적 깨짐 위험 없이 해당 회차의 JSON만 교체하면 되며, 신규 회차 추가 시 새로운 JSON을 추가하고 목록 경로만 바인딩하면 됩니다.
  - **다중 과목 확장성**: 에너지기능사, 공조냉동 등 타 과목을 점진적으로 대량 탑재할 때 프로젝트 디렉토리 구조가 파일별로 깔끔하게 정리됩니다 (`/data/gas/2017_1.json`, `/data/energy/2025_1.json` 등).
* **단점**:
  - **HTTP 요청 수 증가**: 회차를 변경할 때마다 매번 로딩 바가 나타나며, 오프라인 환경 등에서 탐색 경험이 매끄럽지 못할 수 있습니다.
  - **통합 풀이 기능의 복잡화**: 여러 회차의 문제를 골고루 섞어 내는 랜덤 시험 기능을 구현하려면, 각각의 파일들을 병렬 비동기(`Promise.all()`)로 연달아 땡겨와 조립하는 과정이 필요하므로 클라이언트 로직이 다소 무거워집니다.

### 💡 결론 및 권장 방향
현재 가스기능사 18개 회차(1,080문제) 전체 크기가 약 1.1MB 정도로, 현대 브라우저의 대역폭과 성능에서는 단일 파일로 로드하는 것이 **가장 매끄러운 퀴즈 풀이 사용자 경험(UX) 및 추가 기능 구현 편의성**을 제공하므로 현재의 단일 파일 구조가 유리합니다.
단, 향후 **에너지기능사, 에너지기능장 등 타 과목의 모든 회차가 실제로 입고되어 데이터가 수십 MB 단위로 비대해질 경우**, 초기 로드 오버헤드가 크게 증가하므로 **"과목별/회차별 분리 구조"**로 전환하는 것이 유지보수와 성능 측면에서 필수적입니다.
